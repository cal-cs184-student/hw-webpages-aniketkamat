Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau’s algorithm is an algorithm that uses linear interpolation to evaluate a Bezier curve given its control points. Given n control points and a parameter t in between 0 and 1, De Casteljau's algorithm computes n-1 linearly interpolated values between the points using the formula (1-t)*p_(i)+ t*p_(i+1) (eg. ith interpolated value is between control points i and i+1). This is one step of the algorithm. For the next step, the interpolated values from the last step become the new control points and a set of n-2 values (since there are n-1 control points) are calculated with interpolated. This can be repeated until the final step where 1 interpolated value is given by the final 2 control points. Doing this for values of t from 0 to 1 and connecting the final points produces the Bezier curve. In this part, I implemented a function that performed a single step of the algorithm by looping over a given vector of n control points and outputting a vector of n-1 interpolated points using a given t and the aforementioned formula.
The default value of t is 0.5, so each interpolated point is halfway between its two corresponding control points (or points from the prior step). The final evaluated point is shown in red in the final step and is in the expected place for t=0.5 (the mean of the control points). The evaluated point is on the completed green Bezier curve. I then moved around the control points to create a slightly different Bezier Curve to see how the parameter t affects the execution of the algorithm.
Since the underlying green Bezier curve is created from all values of t (from 0 to 1), each final evaluated point (in red) is on the curve. Lower values of t produce final points closer to the first control point on the left while higher values produce final points closer to the last one on the right. When t is maximized, this final evaluated point is exactly the final control point. This is because when you interpolate between two points, p_(i) and p_(i+1), using the formula from above with t=1, the final value is p_(i+1), so each interpolated value at each step is a control point (ie. the first step returns the last n-1 control points, second returns last n-1... last returns last control point).

Part 2: Bezier surfaces with separable 1D de Casteljau

De Casteljau’s algorithm can be extended to Bezier surfaces by looping/interpolating across rows first and then interpolating across those values. It uses the same repeated linear interpolation system as above, but each surface takes in 16 control points (4x4 grid). Each row of this grid is evaluated using de Casteljau’s algorithm across different u values (where u is comparable to the parameter t from part 1). These 4 points are then evaluated using de Casteljau’s algorithm across different v values (also comparable to t), which produces the final point. Overall, I treated each row of 4 control points as a 1D curve and then treated the points produced by each of the 4 rows as a curve which produces the final point. I implemented this by first writing a helper function to run de Casteljau’s algorithm and then doing the following: calling the function on each row of control points (with u) and storing the intermediate points, then calling the function on those points (with v), and returning the final point. As seen above, the extended algorithm can be used to render 3D structures such as the teapot. While it is harder to see each step of the algorithm here, the same system was used to create this image.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

I implemented the area-weighted vertex normals by using a do-while loop to visit each face adjacent to a vertex (that wasn't a boundary). For each face, I got the vertices using the half-edge data structure and used a cross product to compute the normal vector (which is already weighted by area since the magnitude of the vector is twice the area). I added together the normal vectors from each face, normalized them, and returned the result.
The images on the left are rendered without vertex normals while the ones on the right are rendered with vertex normals, resulting in the right images looking smoother/more realistic. One of the most noticeable differences with the right ones (Phong shading) is that the shadows are more natural/connected while the left one produces a more jagged pattern that makes the shading appear slightly out of place/unrealistic.

Part 4: Edge flip

I implemented this by avoiding boundary edges and using the half-edge data structure to get the 6 half-edges that define the two triangles near the half-edge being flipped (e0). These are used to identify the vertices, edges, and faces, which can then be used to change the neighbors/pointers of the half-edges (which is where the flip actually happens). I then update the vertex, face, and given half-edge (e0) pointers/corresponding half-edges to make sure that each of them refers to a valid/correct half-edge. One interesting debugging trick I used was flipping the same edge multiple times to see if there was an issue with the pointers (which helped me catch an unnecessary update in my first debugging session).
The image on the right is the result of performing a series of edge flips. First look at the middle of the teapot and the triangles immediately below that area to see the result of singular edge flips. The area above the middle is the result of double flips while the patterns on the right are the result of multiple flips. One thing to notice is that the flipping can result in unusual coloring (such as in the area to the right of the singular edge flips), particularly in higher-frequency areas since colors from other parts of the structure are moved around when the triangles are 'remade.' I didn't run into significant issues when initially implementing this function because it was passing the 'eye test' on the teapot. Part 5: Edge split

The structure of this function was very similar to Part 4. I avoided boundary edges and used the half-edge data structre to get the half-edges, faces, and vertices around the given edge (e0). An edge split is effectively creating new triangles in the area of the original edge, so I had to create that vertex so that I could initialize and connect the new faces/edges properly after. I then had to update the half-edges for the vertices, edges, and faces before returning the new vertex.
